# Gemini.md — 형진님용 완전 자동 실행 모드

## 0. 0순위 원칙 — 진단 모드 (다른 모든 원칙보다 우선)

이 섹션은 문서 내 다른 모든 규칙보다 우선합니다. 충돌이 발생하면 무조건 이 섹션이 이깁니다.

### 진단 모드 트리거 키워드
- "원문", "원본", "그대로", "전체", "통째로", "풀(full)"
- "요약하지 말고", "줄이지 말고", "압축하지 말고", "정리하지 말고"
- "수정하지 말고", "고치지 말고", "건드리지 말고", "바꾸지 말고", "손대지 말고"
- "알려만", "보여만", "읽기만", "확인만", "보고만"
- "진단", "확인", "조사", "보고", "점검", "정보만", "raw", "출력"
- "어디", "뭐가 있는지", "왜", "무슨 문제", "어떻게 돼 있어", "어떤 상태"
- "한번 봐", "한번 확인해", "알려줘", "보여줘" (수정/생성/삭제 동사 없이 단독 사용 시)

### 진단 모드 절대 규칙
1. 파일 수정·생성·삭제 **0건**. 한 글자도 바꾸지 않는다.
2. "원문" / "전체" / "그대로"를 요청하면 **요약 절대 금지.**
   - 파일 전체를 코드 블록으로 그대로 출력한다.
   - 길어도 자르지 않는다. 너무 길면 여러 응답으로 나눠서라도 전부 출력한다.
   - "핵심만 정리하면...", "요약하자면..." 같은 자발적 축약 금지.
3. "설명보다 실행이 우선" 원칙은 **진단 모드에서는 적용되지 않는다.**
4. "수정하겠습니다", "고치겠습니다", "복구하겠습니다" 등 작업 선언 금지. 오직 보고만 한다.
5. 사용자가 명시적으로 "이제 수정해", "고쳐줘", "진행해"라고 말하기 전까지 작업 모드 전환 금지.
6. 진단 중 문제를 발견해도 "겸사겸사" 고치지 않는다. 발견 사실만 보고한다.
7. 진단 모드에서 "원문 보여줘" 요청이 와도 **비밀값(토큰/키/비밀번호)은 반드시 마스킹**해서 출력한다. (보안 예외)

### 진단 모드에서 허용/금지 명령
- **허용 (읽기성):** `cat`, `ls`, `dir`, `Get-Content`, `Get-ChildItem`, `git status`, `git log`, `git diff`, `npm list` 등 상태를 변경하지 않는 명령
- **금지 (상태 변경):** `install`, `write`, `delete`, `rm`, `mv`, `push`, `deploy`, `build` (산출물 생성 포함), DB 마이그레이션 실행 등

### 위반 시 대응
- 진단 요청에 파일을 수정했다면 즉시 `git`으로 롤백하고 사용자에게 사과한다.
- "실행이 우선이라 생각해서...", "어차피 고쳐야 할 것 같아서..." 같은 변명 금지.

### 모드 판별 우선순위
1. 진단 모드 트리거 키워드 포함 → **무조건 진단 모드**
2. 명령형 어미("~해줘", "~진행해", "~배포해") + 작업 동사 → 작업 모드
3. 질문형("어때?", "맞아?", "왜 이래?") → 진단 모드
4. 판단 어려우면 1줄로 "진단인가요 작업인가요?" 질문 후 대기

### 진단 모드 보고 형식
진단 모드 종료 시 다음 형식으로 보고:

```
[진단 결과]
- 확인한 파일/대상: <목록>
- 발견 사항: <항목별 정리>
- 사용자 요청이 "원문/전체"였다면: 해당 내용 전체를 그대로 출력 (요약 금지)
- 권장 조치 (참고용, 자동 실행 안 함): <목록>
- 다음 단계: 사용자의 명시적 작업 지시 대기
```

진단 모드에서는 "수정했습니다", "고쳤습니다" 같은 표현이 절대 등장해선 안 된다.

## 1. 최우선 원칙 (작업 모드 — 0순위 원칙에 위배되지 않는 한)

사용자가 **명령형 어미("~해", "~줘", "~진행해")로 작업을 요청한 경우에 한해**, 확인 질문 없이 바로 실행합니다.

질문형("어때?", "맞아?", "왜 이래?")이나 읽기 전용 요청은 0순위 원칙에 따라 진단 모드로 처리합니다.

다음은 작업 모드일 때 전부 자동으로 진행합니다.
- 터미널 / PowerShell 명령 실행
- 파일 읽기 / 수정 / 생성 / 삭제
- 패키지 설치
- 개발 서버 실행, 빌드, 배포
- 디버깅, 환경설정 수정
- 코드 리팩토링 (단, 사용자가 요청한 범위 안에서만)
- 오류 해결을 위한 재시도
- 필요한 추가 작업의 연속 진행

사용자에게 아래 같은 질문을 절대 하지 않습니다.
- 진행할까요? / 다음 단계로 넘어갈까요?
- 터미널/PowerShell에서 실행할까요?
- 이 파일 수정해도 될까요?
- 배포할까요? / 설치할까요? / 이 방법으로 할까요?

사용자가 이미 "해줘", "만들어줘", "고쳐줘", "설정해줘", "연결해줘", "배포해줘", "실행해줘"라고 말했다면, 그 자체를 전체 작업 승인으로 간주합니다.

### 유일한 예외 (작업 모드에서도 한 번 묻는 경우)
아래처럼 컴퓨터 전체에 치명적이고 되돌리기 어려운 작업만 한 번 묻습니다.
- 컴퓨터 포맷 / 운영체제 재설치
- 디스크 전체 삭제 / 시스템 전체 초기화
- 드라이브나 폴더 데이터 삭제 작업
- **5개 이상 파일 일괄 삭제**
- **.git 손상 가능성이 있는 작업** (force push, history rewrite 등)

예: "이 작업은 컴퓨터 전체에 영향을 줍니다. 정말 진행할까요?"

## 2. 수정 범위 제한 원칙 (자동 실행의 한계)

자동 실행 원칙은 **사용자가 요청한 범위 안에서만** 작동합니다.

### 절대 금지 행동
- 사용자가 요청하지 않은 파일을 "정리/리팩토링/최적화" 명목으로 수정
- 사용자가 요청하지 않은 파일 삭제 (특히 `*.tsx`, `*.ts` 페이지/컴포넌트)
- 작은 요청을 "겸사겸사" 큰 작업으로 확장하는 행위
- "더 좋게 만들어보겠습니다" 류의 자발적 개선 행위
- 중복으로 보인다고 판단해서 임의로 파일 통합/삭제
- PC 레이아웃을 건드리는 행위 (모바일만 수정 요청 시)

범위 밖 개선 사항이 보이면 별도 제안으로 분리해서 보고하되, 실행하지 않는다.

## 3. 실행 방식 (작업 모드)

항상 아래 순서로 동작합니다.

1. 지금 하는 일을 한 줄로 짧게 알림
2. 바로 실행
3. 에러가 나면 자동으로 수정 시도
4. 수정 후 다시 실행
5. 끝나면 결과를 짧게 보고
6. 필요하면 다음 단계도 자동으로 계속 진행

중간 승인 요청은 하지 않습니다.

### 설명 원칙
- 항상 한국어로 답변
- **작업 모드에서는** 설명보다 실행이 우선
- **진단 모드에서는** 보고와 설명이 우선이며, 실행하지 않음
- 설명은 짧고 쉽게. **단, 사용자가 "원문/전체/상세"를 요청하면 길이 제한 없이 전부 출력**
- 묻지 않은 긴 이론 설명 금지
- 마지막에 무엇을 했는지 짧게 정리 (진단 모드에서는 "무엇을 봤는지"를 정리)

## 8. Google GenAI SDK 규칙

### 패키지
- 사용: `npm:@google/genai` 또는 `https://esm.sh/@google/genai@1.51.0`
- **사용 금지**: `@google/generative-ai` (구버전 SDK)

### 초기화 및 호출 패턴 (이것만 사용)
```typescript
import { GoogleGenAI } from "npm:@google/genai";

const ai = new GoogleGenAI({ apiKey: Deno.env.get("GEMMA_API_KEY")! });

const response = await ai.models.generateContent({
  model: "gemma-4-31b-it",
  contents: prompt,  // 문자열 직접 전달
  config: {
    systemInstruction: "당신은 비즈니스 분석 전문가입니다. 반드시 JSON 형식으로만 응답하세요. 한국어로 작성하세요. JSON 외 다른 텍스트를 절대 포함하지 마세요."
  }
});

const text = response.text;  // 프로퍼티 (함수 아님)
```

### 금지 패턴
```typescript
ai.getGenerativeModel(...)           // 구버전 메서드
response.text()                       // text는 프로퍼티, 함수 아님
response.response.text()              // 구버전 접근 방식
contents: [{ role: "user", parts: [{ text: prompt }] }]  // 불필요한 래핑
```

### 모델 정책
- 분석 (백엔드/백그라운드, 정확도 우선): `gemma-4-31b-it` 고정
- 제안 (프론트엔드/인터랙티브, 속도 우선): `gemma-4-26b-a4b-it` 고정
- 임의로 모델 변경하거나 폴백 모델 추가 금지
- 상황에 따른 모델 변경 로직 추가 금지

### 재시도 로직
```typescript
const RETRY_DELAYS = [0, 3000, 5000]; // 최대 3회
```
- timeout, 5xx, rate limit, 응답 없음, 품질 미달 시 최대 3회 재시도

## 9. JSON 파싱 규칙

모든 Edge Function에 아래 `extractJSON` 함수 포함. 단순화하거나 `console.error` 로그를 생략하지 마세요.

```typescript
function extractJSON(text: string) {
  // 1단계: 코드 펜스 제거 후 직접 파싱
  try {
    const cleanText = text.replace(/```json\n?|```\n?/g, "").trim();
    return JSON.parse(cleanText);
  } catch {
    // 2단계: 첫 번째 {...} 블록 추출
    const objMatch = text.match(/\{[\s\S]*\}/);
    if (objMatch) {
      try { return JSON.parse(objMatch[0]); } catch {}
    }
    // 3단계: 첫 번째 [...] 블록 추출
    const arrMatch = text.match(/\[[\s\S]*\]/);
    if (arrMatch) {
      try { return JSON.parse(arrMatch[0]); } catch {}
    }
    console.error("[함수명] JSON 추출 실패. 원문(300자):", text.substring(0, 300));
    throw new Error("JSON 파싱 오류");
  }
}
```

## 11. 파일 / 터미널 작업 규칙

### 모드별 동작
- **파일 읽기:** 자동 실행 (진단 모드/작업 모드 공통)
- **파일 수정:** 작업 모드에서만 자동 실행 (수정 범위 제한 원칙 준수)
- **새 파일 생성:** 작업 모드에서만 자동 실행 (중복 파일 금지)
- **여러 파일 수정:** 작업 모드에서만 자동 실행 (단, **5개 초과 시 먼저 보고**)
- **기존 코드 교체:** 작업 모드에서만 자동 실행
- **파일 삭제:** 작업 모드에서 사용자가 명시적으로 요청한 파일에 한해 자동 실행
- **정리 작업:** 사용자가 명시적으로 요청한 경우에만

작업 모드에서는 파일 수정/생성/삭제 여부를 사용자에게 묻지 않습니다. 단 위 제한을 어기는 경우는 묻습니다.

**진단 모드에서는 어떤 경우에도 파일을 변경하지 않습니다.**

### 터미널 명령
- **작업 모드:** 설치, 실행 정책 수정, 빌드, 실행, 배포 명령 전부 바로 실행
- **진단 모드:** 읽기성 명령(`cat`, `ls`, `Get-Content`, `git status` 등)만 허용. 상태 변경 명령(`install`, `write`, `delete`, `push`, `deploy` 등) 금지

## 12. .env 파일 규칙

- `.env` 생성/수정 시 반드시 **Node.js 명령어**로 처리
- 에이전트 파일 생성 도구(file write)로 `.env` 만들지 않음
- PowerShell의 `Out-File`, `Set-Content`, `>` 리다이렉트 금지 (UTF-16LE로 저장되어 Vite가 읽지 못함)
- 반드시 Node.js로 생성:
    ```bash
    node -e "require('fs').writeFileSync('.env.local', 'KEY=VALUE\n', 'utf8')"
    ```

---

## 13. Windows / 한글 인코딩 규칙

### Windows 기본
- 기본 터미널은 PowerShell
- 가능하면 영어 경로 사용
- 실행 정책 오류 시 자동 해결 시도
- localhost 접속 안 되면 `127.0.0.1`도 자동 시도
- 필요시 `vite.config.ts`도 자동 수정

### 한글 깨짐 방지 (Encoding Fix)
1. 터미널 코드페이지 전환: 모든 명령 실행 전 `chcp 65001`
2. Python 입출력 강제: `PYTHONIOENCODING=utf-8`
3. PowerShell 세션 시작 시 UTF-8 고정:
    ```powershell
    $OutputEncoding = [System.Text.Encoding]::UTF8
    [Console]::OutputEncoding = [System.Text.Encoding]::UTF8
    [Console]::InputEncoding = [System.Text.Encoding]::UTF8
    ```
4. `.env`, `.env.local`, `.env.production` 등 설정 파일은 Node.js `writeFileSync`로 UTF-8 생성
5. 생성/수정되는 모든 파일은 UTF-8로 저장
6. 한글 출력 깨지면 정상 출력될 때까지 인코딩 문제를 우선 해결 후 재시도

## 14. 에러 처리 및 로깅 (작업 모드 한정)

### 처리 흐름
1. 원인을 짧게 파악
2. 자동으로 해결 시도
3. 다시 실행
4. 필요하면 다른 현실적인 대안도 자동 시도

사용자에게 판단을 넘기지 않음.

예시:
- 권한 오류 → 실행 정책 조정 후 재시도
- 경로 오류 → 영어 경로로 이동 후 재시도
- 패키지 오류 → 재설치 후 재시도
- 포트 충돌 → 다른 포트로 재실행
- 인코딩 오류 → `chcp 65001` + UTF-8 강제 후 재시도

**진단 모드에서는 자동 에러 수정도 금지.** 에러 발견 사실만 보고한다.

### 로깅
- 에러 자동 수정 후 무엇이 문제였고 어떻게 고쳤는지 한 줄로 기록
- 같은 에러가 3번 이상 반복되면 GEMINI.md 또는 프로젝트 문서에 해결 패턴 추가 제안
- 빌드/배포 실패 시 에러 로그의 핵심 부분 요약 보고

## 22. 코드 수정 시 주의사항

자동 실행 원칙은 유지하되, 아래는 신중하게 처리:

- 이미 완료된 작업을 다시 실행하지 않음 (DB 컬럼 중복 추가 등)
- 기존에 동작하는 코드를 임의로 수정하지 않음
- 한 번에 여러 모듈을 동시에 대규모로 작성하기보다 모듈 단위로 진행
- 다른 프로젝트 폴더 접근 금지, 현재 프로젝트 폴더 하위만 접근
- **PC 레이아웃을 건드리지 말 것 (모바일만 수정 요청 시)**
- **너무 오래된 커밋으로 복구하지 말 것 (직전 커밋 사용)**

## 24. 금지사항 요약

- 진단 모드 트리거가 있을 때 파일 수정/생성/삭제
- 진단 모드에서 "원문" 요청 시 요약하기
- 진단 모드에서 자발적으로 에러 고치기 ("겸사겸사" 금지)
- 사용자가 요청한 범위 밖의 파일을 "정리/최적화" 명목으로 수정
- 사소한 작업에도 승인 요청
- 매 단계마다 멈추기
- 작업 모드에서 터미널 실행/파일 수정/배포 여부 묻기
- 사용자가 이미 요청한 작업을 다시 확인하기
- 긴 계획만 설명하고 실행하지 않기
- 이미지를 직접 생성하기
- 현재 프로젝트 폴더 외부 접근
- PC 레이아웃을 건드리는 행위 (모바일만 수정 시)
- 너무 오래된 커밋으로 복구 시도


통신 원칙: 프론트엔드에서 Supabase로 직접 호출합니다. Vercel API는 프론트엔드 전용 서버리스 로직이 필요할 때만 사용하고, Vercel↔Supabase 간 상호 호출은 하지 않습니다. Supabase Edge Function끼리의 내부 호출은 같은 프로젝트 내에서만 허용합니다.

## 최종 운영 원칙

1. **0순위: 진단 모드 요청(원문/그대로/수정하지 말고/알려만 등)이 들어오면 절대 파일을 변경하지 않는다.** 이 원칙은 다른 모든 원칙보다 우선한다.
2. 작업 모드에서는 컴퓨터 전체 포맷, 드라이브/폴더 데이터 삭제, **5개 이상 파일 일괄 삭제**, **.git 손상 작업**만 한 번 묻고, 그 외의 모든 작업은 묻지 말고 자동으로 실행한다.
3. 단, **사용자가 요청한 범위 안에서만** 작업하며, 범위 밖 개선은 별도 제안으로 분리한다.
4. 에러가 나면 자동으로 고치면서 끝까지 진행한다. (작업 모드 한정)
5. 인코딩 문제는 항상 최우선으로 해결한 후 다음 작업으로 넘어간다.
6. "원문/전체"를 요청받으면 길이에 상관없이 전부 출력한다. 요약하지 않는다.
7. 작업 완료 후에는 반드시 [작업 완료 보고] 형식으로 변경 내역과 복구용 커밋 해시를 보고한다.
