# 내친구 케이(K-bestie) — 1차 개발 착수 명세서

문서 목적: Claude Code 기반 개발 착수를 위한 범위·사양 전달 기준 레퍼런스: Lovable 프로토타입 (heartbloom-ai-pal.lovable.app) 작성 기준일: 2026-06-07

---

## 1. 개발 범위 및 기술 스택

Lovable 프로토타입에 구현된 모든 화면과 기능을 개발 대상으로 한다. 단, Lovable은 기능·UI 레퍼런스일 뿐이며, 실제 구현은 Claude Code를 활용해 우리 확정 스택 위에 새로 개발한다.

확정 스택은 프론트엔드 React + Next.js(PWA), 백엔드·DB는 Supabase(PostgreSQL), 배포는 Vercel이다. 실시간 음성 대화는 Gemini Live API, 백그라운드 리포트 생성은 Gemma 계열 모델을 사용한다.

지켜야 할 고정 제약은 세 가지다. 기존 13개 DB 테이블과 스키마는 변경하지 않는다. API 키는 프런트에 노출하지 않으며 ephemeral token 방식으로 처리한다. 과도한 자동화는 지양한다.

---

## 2. 1차 개발의 핵심 원칙 — "심장 먼저, 곁가지는 placeholder"

1차 개발은 "음성 대화 → 1분 요약 리포트 → 부모 PWA 확인"이라는 핵심 가치 사슬을 끊김 없이 작동시키는 데 집중한다. 이 한 사슬이 서비스의 심장이며, MVP의 최소 기능 정의다.

가입·로그인·법정대리인 동의·결제 등 곁가지는 1차에서 더미(교체 가능한 placeholder)로 두되, 나중에 실제 기능으로 교체하기 쉽게 분리해 구현한다. 가장 어렵고 시간이 많이 드는 핵심(음성 파이프라인, 케이 대화 프롬프트, 리포트 생성)은 한 번만 제대로 만들고, 이후 곁가지를 이어 붙이는 방식으로 확장한다.

케이의 대화 설계 프롬프트와 리포트 요약 프롬프트는 코드와 분리된 독립 자산(별도 파일·버전 관리)으로 관리한다. 이 두 프롬프트가 케이의 핵심 자산이므로, 모델이나 화면이 바뀌어도 재사용 가능해야 한다.

---

## 3. 5가지 핵심 결정사항 (확정)

첫째, 부모의 리포트 수신 경로는 내친구 케이 PWA 접속이다. 별도 알림톡·문자·이메일 발송이 아니라, 부모가 PWA에 들어와 리포트를 확인한다. Gate A의 핵심 지표인 "부모의 자발적 재방문"을 측정하려면 PWA 접속·열람 로그를 남겨야 한다.

둘째, 음성 모델은 Gemini 3.1 Flash Live 하나로 고정해 시작한다. 다만 app/api/_lib/ai.ts에 모델 스위치 구조(모델 ID·config 매핑)를 만들어두어, 추후 2.5 Flash Live로 전환할 때 호출 코드를 건드리지 않고 모델만 교체할 수 있게 한다.

셋째, 리포트 형태와 아이 입력(온보딩) 화면은 Lovable 구현분을 그대로 개발한다(상세는 5절·6절).

넷째, 위험 신호 처리(전문가 연결 포함)는 Phase 2로 미룬다. Lovable에 '전문가와 연결하기' 화면이 있으나, 트리거 로직과 기관 연계는 전문가·법률 검토가 끝난 뒤 구현한다(상세는 8절).

다섯째, 부모 질문 등록 → 케이가 아이에게 질문 기능은 Lovable에 구현돼 있으므로 1차 개발 범위에 포함한다(상세는 7절).

## 4. 아이 음성 대화 사양 (P3, 핵심 MUST)

음성 파이프라인은 폰 브라우저에서 마이크 입력을 받아 16kHz PCM으로 리샘플하고, WebSocket으로 Gemini Live API에 전송한 뒤, 24kHz 오디오로 재생한다. 오디오 시작은 반드시 사용자 탭 제스처에서 시작한다(iOS 제약). 세션은 15분 제한이며 session resumption·context compression으로 연장한다. 인증은 백엔드가 발급하는 ephemeral token으로 처리한다.

대상 환경은 iOS Safari 최신 + Android Chrome 최신이며 HTTPS 필수다. iPhone에서 Chrome은 지원하지 않고 Safari로 안내한다.

케이의 대화 설계는 다음 원칙을 따른다. 고정 질문이 아니라 아이 관심사로 시작해 라포를 형성한 뒤 일상·감정 질문으로 넘어간다. 케이의 모든 발화는 "공감 한 마디 + 질문 하나"로 끝내며, 맞장구만 치고 멈추지 않는다. 발화는 최대 두 문장, 짧고 쉬운 말로 한다. 아이가 방금 한 말을 먼저 받아 확인한 뒤 질문하고, 시제(할 거야/했어)를 구분하며, 이미 말한 걸 다시 묻지 않는다. 추궁·훈계·평가는 금지하고 공감하며 듣는다.

산만한 환경 대응은 세 겹으로 구성한다. VAD 민감도를 LOW로 설정하고(start_of_speech_sensitivity LOW, silence_duration_ms 800ms 이상), 케이 발화를 짧게 강제하며, 화면 탭으로 케이 발화를 수동 중단해 아이 말하기 모드로 전환한다. 아이가 말을 시작하면 케이가 즉시 멈추는 barge-in을 적용한다.

음성 질문은 화면 자막과 동시에 표시한다. 아이가 음성을 흘려도 글자로 다시 읽고 답할 수 있게 하기 위함이며, Live API의 output transcription 기능을 켜서 구현한다.

화자 식별(특정 아이 목소리만 인식)은 기술적으로 불가하므로 범위에서 제외한다(WON'T).

## 5. 부모용 리포트 사양 (P1, 핵심 MUST)

리포트는 부모가 출퇴근길 1분 내에 읽을 수 있는 아이 하루 요약이다. Lovable 화면 기준으로, Gemma 요약 엔진이 채워야 할 출력 항목은 다음과 같다.

아이 이름과 오늘 하루 한 줄 요약(예: "친구와 사소한 다툼이 있었지만 잘 풀어낸 하루였어요. 체육 시간을 가장 좋아했어요."), 기분 점수(예: 7/10), 오늘 대화 횟수(예: 2회), 마지막 대화 시점(예: 35분 전), 감정 태그 복수 개(예: #즐거움 #약간의 짜증 #뿌듯함 #호기심), 그리고 부모용 오늘의 대화 가이드(부모가 아이에게 건넬 한 마디 추천, 예: "서준이가 오늘 학교에서 가장 즐거웠던 순간이 뭔지 슬쩍 물어보세요.")다.

개발 시 Gemma는 위 항목을 채우는 구조화된 출력(JSON 등)을 반환하도록 프롬프트를 설계하고, 프론트는 그 구조를 받아 Lovable과 동일한 카드 UI로 렌더링한다. 리포트 생성은 실시간 대화와 분리된 백그라운드 처리이며, 대화 종료 후 60초 이내 생성을 목표로 한다.

프라이버시 원칙(영구): 부모는 아이-케이 대화 원문을 전체 열람할 수 없다. 리포트는 요약·인사이트만 제공하며, DB 레벨에서 대화 원문 테이블은 부모 역할로 조회 불가하도록 차단한다(Supabase RLS). 요약 생성 파이프라인만 system 권한으로 원문에 접근한다.

리포트 하단 "AI 분석은 참고용이며 전문적인 진단·치료를 대체하지 않습니다" 고지 문구는 Lovable과 동일하게 유지한다(법적·윤리적 안전장치).

---

## 6. 아이 입력(온보딩) 사양

가입은 1차에서 더미(placeholder)로 두지만, 아이 정보 입력은 실제로 받는다(Lovable 구현 기준). 입력 항목은 아이 이름, 학년, 관심사 키워드(예: 로블록스 'fly hide and seek', 크레스티드 게코 '효효' 등)다. 이 관심사는 케이 대화의 진입점으로 쓰이므로 대화 프롬프트에 주입되어야 한다.

향후 DB 연동으로 쉽게 교체할 수 있도록, 아이 정보를 읽어오는 통로는 데이터 소스와 분리해 구현한다. 1차에서는 입력값을 Supabase에 저장하되, 가입·계정 연결 부분만 더미로 둔다.

## 7. 부모 질문 등록 → 케이 대리 질문 기능

이 기능은 부모가 궁금한 점을 직접 묻기 어려울 때, 케이를 통해 자연스럽게 묻게 하는 소통 브릿지 기능이다. Lovable의 /parent/guide 화면에 구현돼 있다.

작동 방식은 이렇다. 부모가 질문을 등록하면(예: "오늘 받아쓰기 시험은 어땠어?") 그 질문이 대기 중 상태로 큐에 쌓인다. 케이는 다음 대화에서 이 질문을 흐름에 맞춰 자연스럽게 한 번 아이에게 묻는다. 전달되면 상태가 전달됨으로 바뀌고, 같은 질문을 반복하지 않는다. 부모는 질문을 중지할 수 있어 중지됨 상태도 존재한다. 각 질문에는 등록 시각, 전달 횟수, 마지막 전달 시각이 표시된다.

개발 시 필요한 데이터는 질문 텍스트, 상태(대기 중/전달됨/중지됨), 등록 시각, 전달 횟수, 마지막 전달 시각이다. 기존 13개 테이블 변경 금지 원칙이 있으므로, 이 질문 큐를 어느 테이블에 어떻게 담을지는 기존 스키마를 확인해 결정한다(스키마 변경이 불가피하면 별도 협의).

케이 대화 프롬프트에는 "부모가 등록한 대기 중 질문이 있으면 대화 흐름에 맞춰 한 번 자연스럽게 물어보고, 전달 후에는 반복하지 않는다"는 규칙을 주입한다.

프라이버시 정합성: 부모가 질문을 등록하는 것은 허용되나, 그 답변 원문을 부모가 그대로 보는 것이 아니라 요약·인사이트 형태로 받는다. 원문 비공개 원칙을 유지한다.

## 8. Phase 2로 미루는 항목 (1차 제외)

위험 신호 처리 로직은 1차에서 제외한다. Lovable의 '전문가와 연결하기'(Wee센터·아동전문기관 연결) 화면은 UI로만 두되, 실제 위험 신호 트리거 기준과 기관 연계 동작은 전문가·법률 검토 후 Phase 2에서 구현한다.

가입·로그인·법정대리인 동의(PIPA)·PASS 본인인증·결제 모듈은 1차에서 더미로 두고 Phase 2에서 정식 결합한다. MVP 정식 단계에서는 이메일+생년월일+약관 동의, Google·카카오 간편 로그인, 18세 미만 자기선언식 차단을 적용하되, 강한 본인인증·법정대리인 동의 고도화는 Phase 2다.

Gemini 2.5 Flash Live 스위치 검증(affective dialog), 완전 텍스트 대화 모드(저렴한 모델 라우팅을 통한 비용 절감), 모드 전환(음성↔채팅·케이 목소리 끄기)은 Could 항목으로 1차 이후 단계에서 다룬다.

## 9. 1차 개발 완료 기준 (Definition of Done)

아이가 PWA에서 관심사로 케이와 음성 대화를 시작하고 완료할 수 있다. 대화 종료 후 Gemma가 1분 요약 리포트(5절 항목)를 생성한다. 부모가 PWA에 접속해 그 리포트를 확인할 수 있고, 재방문·열람이 로그로 측정된다. 부모가 등록한 질문이 케이의 다음 대화에 반영된다. 산만한 환경 대응 3종(VAD LOW, 짧은 발화, 화면 탭 중단)과 자막이 동작한다.

이 상태가 곧 내부 5가족 테스트(Gate A: 부모 5명의 자발적 리포트 재방문 검증) 착수가 가능한 데모 기준선이다.
